# L2 Requirements — GetFiles CLI Tool

---

## L1-1: .NET Tool Packaging and Project Structure

### L2-1.1: Solution and Project Setup

The solution shall contain a `GetFiles.sln` solution file and a `src/GetFiles` project directory with a `GetFiles.csproj` file targeting `net8.0`. The `.csproj` shall declare `<PackAsTool>true</PackAsTool>`, `<ToolCommandName>gf</ToolCommandName>`, and `<PackageId>QuinntyneBrown.GetFiles</PackageId>` with appropriate NuGet metadata (author, description, license, etc.).

**Acceptance Criteria:**
- AC-1: The `.csproj` file contains `PackAsTool`, `ToolCommandName`, and `PackageId` properties with the correct values.
- AC-2: `dotnet build` succeeds without errors.
- AC-3: `dotnet pack` produces a `.nupkg` file.

### L2-1.2: Dependency Injection and Hosting

The application shall use `Microsoft.Extensions.DependencyInjection` to register all services. `Microsoft.Extensions.Logging` shall be configured with console logging. `Microsoft.Extensions.Configuration` shall be wired for optional `appsettings.json` support. The composition root shall be in `Program.cs`.

**Acceptance Criteria:**
- AC-1: `Program.cs` configures a `ServiceCollection` with logging, configuration, and application services.
- AC-2: All command handlers receive dependencies via constructor injection.
- AC-3: Console logging is functional and outputs at configurable verbosity levels.

### L2-1.3: System.CommandLine Integration

The CLI shall use `System.CommandLine` to define a root command with subcommands. Each command shall be defined in its own file following the command-per-file pattern (e.g., `Commands/AggregateCommand.cs`).

**Acceptance Criteria:**
- AC-1: The root command provides `--help` output describing the tool.
- AC-2: Each subcommand is defined in a separate file under a `Commands` directory.
- AC-3: `System.CommandLine` parses arguments and options correctly, returning non-zero exit codes on invalid input.

### L2-1.4: Default Command

When no subcommand is supplied, the CLI shall behave as if the `aggregate` command had been specified, so that `gf -p ./repo` is equivalent to `gf aggregate -p ./repo`. Help (`-h`, `--help`, `-?`) and version (`--version`) flags shall be passed through to the root command unchanged, and the explicit `gf aggregate ...` form shall continue to work.

**Acceptance Criteria:**
- AC-1: `gf -p ./repo` runs the aggregate command and produces an output file.
- AC-2: `gf aggregate -p ./repo` continues to work identically (backward compatible).
- AC-3: `gf --help` and `gf --version` display root help/version rather than being interpreted as aggregate options.
- AC-4: Invoking `gf` with no arguments displays root help.

## L1-2: Repository File Discovery and Filtering

### L2-2.1: File Discovery by Extension

The tool shall recursively walk the provided repository path and collect files matching Angular extensions (`.ts`, `.html`, `.scss`, `.css`) and C# extensions (`.cs`, `.csproj`, `.sln`). Configuration files (`.json`, `.yaml`, `.yml`) in the project shall also be included. The set of included extensions shall be defined in a central location for maintainability.

**Acceptance Criteria:**
- AC-1: Given a directory containing `.ts`, `.cs`, `.html`, `.scss`, `.css`, `.csproj`, `.sln`, and `.json` files, all are discovered.
- AC-2: Files with extensions not in the inclusion list (e.g., `.exe`, `.dll`, `.png`) are excluded.
- AC-3: Discovery is recursive through subdirectories.

### L2-2.2: .gitignore Filtering

The tool shall parse `.gitignore` files found in the repository root (and nested `.gitignore` files) and exclude any files or directories matching the ignore patterns. If no `.gitignore` file exists, no gitignore-based filtering is applied.

**Acceptance Criteria:**
- AC-1: Files matched by `.gitignore` patterns are excluded from the output.
- AC-2: Nested `.gitignore` files are respected relative to their directory.
- AC-3: If no `.gitignore` exists, all matching-extension files are included.

### L2-2.3: Hardcoded Directory Exclusions

The tool shall unconditionally exclude `node_modules`, `dist`, `bin`, `obj`, and `.git` directories from file discovery, regardless of `.gitignore` content.

**Acceptance Criteria:**
- AC-1: Files inside `node_modules/` are never included.
- AC-2: Files inside `dist/` are never included.
- AC-3: Files inside `bin/`, `obj/`, and `.git/` are never included.

### L2-2.4: User-Specified Ignore Paths

The `aggregate` command shall accept an `--ignore` option (alias `-i`) that can be specified multiple times. Each value is a folder or path pattern to exclude from file discovery, in addition to the hardcoded exclusions and `.gitignore` rules.

**Acceptance Criteria:**
- AC-1: `gf -p ./repo --ignore tests --ignore docs` excludes files in `tests/` and `docs/` directories.
- AC-2: Multiple `--ignore` values can be specified in a single invocation.
- AC-3: Ignore paths work at any depth in the directory tree (e.g., `--ignore migrations` excludes `src/db/migrations/`).
- AC-4: When no `--ignore` options are provided, behavior is unchanged from default.
- AC-5: The `--ignore` option is also accepted as `-i`.

## L1-3: Code Aggregation and Output

### L2-3.1: Aggregate Command

The tool shall provide an `aggregate` subcommand that accepts a required `--path` option (alias `-p`, the repository path) and an optional `--output` option (alias `-o`, the output file path, defaulting to `codebase.txt` in the current directory). As the default command, `aggregate` may be omitted from the invocation.

**Acceptance Criteria:**
- AC-1: `gf aggregate --path ./my-repo` produces an output file.
- AC-2: `gf aggregate --path ./my-repo --output result.txt` writes to `result.txt`.
- AC-3: If `--path` is omitted, the CLI reports an error with usage guidance.
- AC-4: The short aliases `-p` and `-o` are accepted (e.g., `gf -p ./my-repo -o result.txt`).

### L2-3.2: Output File Format

Each file in the output shall be delimited with a header showing the file's relative path from the repository root, followed by the file content, followed by a separator. The format shall be:

```
=== FILE: relative/path/to/file.cs ===
<file content>
=== END FILE: relative/path/to/file.cs ===
```

**Acceptance Criteria:**
- AC-1: Each file has a header line containing `=== FILE:` and its relative path.
- AC-2: Each file has a footer line containing `=== END FILE:` and its relative path.
- AC-3: The relative path uses forward slashes regardless of OS.

### L2-3.3: Summary Statistics

After aggregation, the tool shall log a summary to the console including: total files processed, total lines of code, and output file size in human-readable format.

**Acceptance Criteria:**
- AC-1: The console output includes the count of files aggregated.
- AC-2: The console output includes the total line count.
- AC-3: The console output includes the output file size (e.g., "1.2 MB").

## L1-4: Comment and Whitespace Stripping

### L2-4.1: Strip Comments Option

The `aggregate` command shall strip comments by default. The `--strip-comments` flag defaults to `true`. A `--no-strip-comments` flag disables it. When enabled, the tool shall remove single-line comments (`//`), multi-line comments (`/* */`), and HTML comments (`<!-- -->`) from file contents before writing to the output.

**Acceptance Criteria:**
- AC-1: C#/TypeScript single-line comments (`// ...`) are removed.
- AC-2: C#/TypeScript multi-line comments (`/* ... */`) are removed.
- AC-3: HTML comments (`<!-- ... -->`) are removed.
- AC-4: String literals containing comment-like patterns are preserved (e.g., `"https://example.com"`).
- AC-5: Comments are stripped by default when `--strip-comments` is not explicitly specified.
- AC-6: `--no-strip-comments` disables comment stripping.

### L2-4.2: Strip Whitespace Option

The `aggregate` command shall strip whitespace by default. The `--strip-whitespace` flag defaults to `true`. A `--no-strip-whitespace` flag disables it. When enabled, the tool shall collapse consecutive blank lines into a single blank line and trim trailing whitespace from each line.

**Acceptance Criteria:**
- AC-1: Three or more consecutive blank lines are collapsed to a single blank line.
- AC-2: Trailing whitespace is removed from each line.
- AC-3: Leading indentation is preserved (only trailing whitespace and blank lines are affected).
- AC-4: Whitespace stripping is on by default when `--strip-whitespace` is not explicitly specified.
- AC-5: `--no-strip-whitespace` disables whitespace stripping.

### L2-4.3: Combined Stripping

When both `--strip-comments` and `--strip-whitespace` are enabled, comments shall be stripped first, then whitespace collapsing applied to the result.

**Acceptance Criteria:**
- AC-1: Using both flags together produces smaller output than either flag alone.
- AC-2: The order of operations is comments first, then whitespace.
