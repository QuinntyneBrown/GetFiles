# L1 Requirements — GetFiles CLI Tool

## L1-1: .NET Tool Packaging and Project Structure

The solution shall be a .NET global/local tool named `get-files` with package ID `QuinntyneBrown.GetFiles`, structured as a proper .NET tool suitable for publishing to NuGet. The project shall use `System.CommandLine` for CLI parsing and Microsoft Extensions (`Microsoft.Extensions.DependencyInjection`, `Microsoft.Extensions.Logging`, `Microsoft.Extensions.Configuration`) for cross-cutting concerns. The CLI shall follow a command-per-file pattern.

## L1-2: Repository File Discovery and Filtering

The tool shall accept a path to a code repository and discover all Angular (`.ts`, `.html`, `.scss`, `.css`, `.json`) and C# (`.cs`, `.csproj`, `.sln`) source files. It shall respect `.gitignore` rules to exclude ignored files. It shall unconditionally exclude `node_modules` and `dist` directories regardless of `.gitignore` content. Users shall be able to specify additional folders/paths to ignore via command-line options.

## L1-3: Code Aggregation and Output

The tool shall aggregate all discovered source files into a single text output file. Each file's content shall be clearly delimited with its relative path so that an LLM (Ollama) can understand the codebase structure. The output shall be well-structured for LLM consumption.

## L1-4: Comment and Whitespace Stripping

The tool shall strip comments and collapse excessive whitespace from the aggregated output by default. This shall support C#-style comments (`//`, `/* */`), TypeScript/JavaScript comments (`//`, `/* */`), and HTML comments (`<!-- -->`). This reduces output size for more efficient LLM context usage. Users may disable stripping via `--no-strip-comments` and `--no-strip-whitespace` flags.
